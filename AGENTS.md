## Общие правила работы

- Перед изменениями изучи существующую архитектуру и не ломай текущие соглашения проекта.
- Не оставляй legacy/compat-код, если задача явно предполагает рефакторинг на новую схему.
- Не добавляй новые production-зависимости без явного обоснования.
- Сохраняй существующие комментарии, если логика рядом не была полностью заменена.
- После изменений кратко опиши:
  - какие файлы изменены;
  - зачем они изменены;
  - какие проверки выполнены;
  - какие риски или ограничения остались.

## Документация

- Если меняется поведение приложения, обнови README, docs или комментарии к env-переменным.
- Для новых env-переменных добавь описание: назначение, значение по умолчанию, пример. Описание располагается над строкой с переменной
- Любая новая или изменяемая функция должны быть подробно задокументированы: что делает функция, какие параметры принимает, для чего и где используется
- Новый или изменяемый функционал должен иметь комментарии по основным этапам выполнения: что происходит, для чего, примеры, если есть

## Код

- Придерживайся хорошей читаемости и качества кода
- Не делай крупные архитектурные перестройки без необходимости или явной инструкции
- При изменении проекта старайся учитывать существующую архитектуру и ее особенности, старайся аккуратно добавлять новую логику
